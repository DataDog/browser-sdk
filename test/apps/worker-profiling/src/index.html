How to instrument a dedicated worker

Step 1 Initialize RUM with a profiling sample rate

Enable profiling on the main thread. Workers are profiled when the session is sampled — no separate rate needed.

main.ts
import { datadogRum } from '@datadog/browser-rum'

datadogRum.init({
  clientToken: '...',
  applicationId: '...',
  sessionSampleRate: 100,
  profilingSampleRate: 100, // workers follow this rate
})
Pattern A Persistent worker

A worker that runs for the lifetime of the page. Register it once on the main thread; the SDK profiles it continuously, flushing a trace every 60 s.

main.ts
const worker = new Worker('/my-worker.js', {
  name: 'my-worker', // optional — appears as tag in Datadog
})

// Returns an unregister function. Call it when done — SDK flushes
// the current profiling session. You still control terminate().
const unregister = datadogRum.registerProfilingWorker(worker, { name: 'my-worker' })

// When you're done with the worker:
unregister()       // flushes current profile session
worker.terminate() // you own the worker lifecycle
my-worker.ts
import { startProfilingWorker } from '@datadog/browser-rum/worker'

// Call once at the top — that's it
startProfilingWorker()

// ... your worker logic ...

⚠️ The worker script URL must be served with Document-Policy: js-profiling.

Pattern B Short-lived worker — self-managed lifecycle

The worker knows when it's done. Call stop() to flush the current profile, then self.close() to exit. You stay in control of the worker lifecycle.

main.ts
const worker = new Worker('/burst-worker.js', { name: 'burst-worker' })

// Register — SDK starts profiling immediately
datadogRum.registerProfilingWorker(worker, { name: 'burst-worker' })

// No cleanup needed — worker manages its own lifecycle
burst-worker.ts
import { startProfilingWorker } from '@datadog/browser-rum/worker'

const stop = startProfilingWorker()

async function run() {
  await doHeavyComputation()

  // Flush the profile, then close the worker yourself.
  await stop()
  self.close()
}

run()
Pattern C Short-lived worker — main-thread lifecycle

The main thread decides when to stop the worker. Call flushAndTerminateProfilingWorker() instead of worker.terminate() — it flushes the profile first, then terminates the worker after a 5 s safety timeout.

main.ts
const worker = new Worker('/task-worker.js', { name: 'task-worker' })
const unregister = datadogRum.registerProfilingWorker(worker, { name: 'task-worker' })

// Later, when you want to stop it:
unregister()       // flushes current profile session
worker.terminate() // you own the worker lifecycle
task-worker.ts
import { startProfilingWorker } from '@datadog/browser-rum/worker'

// Nothing special — just start and do work
startProfilingWorker()

// ... your worker logic ...
// Main thread calls unregister() + worker.terminate() when done
Requirement HTTP header on the worker script

The worker script response (not just the document) must include this header, otherwise new Profiler() throws a NotAllowedError. 

Document-Policy: js-profiling

Express / webpack-dev-server example:

// Express
app.get('/my-worker.js', (req, res, next) => {
  res.setHeader('Document-Policy', 'js-profiling')
  next()
})

// webpack-dev-server (devServer.headers only sets it on the document)
// — use setupMiddlewares for the worker route specifically
